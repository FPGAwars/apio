# Apio api

---

## apio api

The command group `apio apio` contains subcommands that are
intended to be used by tools and programs such as Icestudio, rather
than being used directly by users.

<h3>Options</h3>

```
-h, --help  Show this message and exit.
```

<h3>Subcommands</h3>

```
apio api get-system
apio api get-project
apio api get-build-report
apio api get-boards
apio api get-fpgas
apio api get-programmers
apio api get-examples
apio api get-commands
apio api scan-devices
```

---

## apio api get-system

The command `apio api get-system` exports information about apio and
the underlying system as a JSON doc. It is similar to the command
`apio info system` which is intended for human consumption.

The optional flag `--timestamp` allows the caller to embed in the JSON
document a known timestamp that allows to verify that the JSON
document was indeed generated by the same invocation.

<h3>Examples</h3>

```
apio api get-system                # Write to stdout
apio api get-system  -o apio.json  # Write to a file
```

<h3>Options</h3>

```
-t, --timestamp text    Set a user provided timestamp.
-o, --output file-name  Set output file.
-f, --force             Overwrite output file.
-h, --help              Show this message and exit.
```

## apio api get-build-report

The command `apio api get-build-report` retrieves the resource
utilization and timing (fmax) metrics from the most recent
`apio build` or `apio report` run and emits them as a structured JSON
document.

This command targets machine-readable pipeline automation and is
completely decoupled from the human-centric output of `apio report`.
Both commands read the same underlying `.pnr` build artifact but serve
distinct audiences.

If no prior build data exists in the workspace, the command prints a
clear error message to stderr and exits with a non-zero status code so
that CI/CD pipelines fail fast rather than silently consuming stale or
empty data.

<h3>Output schema</h3>

```json
{
  "timestamp": "<optional user string>",
  "build-report": {
    "env-name": "default",
    "build-dir": "_build/default",
    "pnr-report-file": "_build/default/hardware.pnr",
    "utilization": {
      "ICESTORM_LC": { "used": 142, "available": 3520 }
    },
    "clocks": {
      "clk$glb_clk": { "fmax": 48.23 }
    }
  }
}
```

The `clocks` field is always present and is always a dict. It is `{}`
when the build produced no clock timing data, so consumers can iterate
over it safely without an existence check.

The fields `board-id`, `fpga-id`, and `bitstream-files` are
intentionally absent to prevent stale project-state mismatches and
unsafe glob-based file discovery at consumption time.

<h3>Examples</h3>

```
apio api get-build-report                 # Write to stdout
apio api get-build-report -e env1         # Use specified env
apio api get-build-report -p foo/bar      # Project in another dir
apio api get-build-report -o report.json  # Write to a file
```

<h3>Options</h3>

```
-e, --env name          Set the apio.ini env.
-p, --project-dir path  Set the root directory for the project.
-t, --timestamp text    Set a user provided timestamp.
-o, --output file-name  Set output file.
-f, --force             Overwrite output file.
-h, --help              Show this message and exit.
```

## apio api echo

The command `apio api echo` allows external programs such as the Apio
VS Code extension to print a short message in a format that is
consistent with the Apio theme currently selected in user preferences.

The required option `--style` should have one of these values: `OK`,
`INFO`, `WARNING`, `ERROR`, `TITLE`, `EMPH1`, `EMPH2`, or `EMPH3`.
The style colors can be viewed with the command `apio info themes`.

<h3>Examples</h3>

```
apio api echo -t "Hello world" -s INFO
apio api echo -t "Task completed successfully" -s OK
apio api echo -t "Task failed" -s ERROR
```

<h3>Options</h3>

```
-t, --text MESSAGE  Set message to echo.  [required]
-s, --style STYLE   Set style to use.  [required]
-h, --help          Show this message and exit.
```
